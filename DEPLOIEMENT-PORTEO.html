en production — gratuitement

Plateforme de gestion de portage salarial : front Angular, API .NET 8 et base PostgreSQL. Déploiement réparti sur Render (API + base) et Firebase Hosting (front), via GitHub.

Front : Firebase Hosting API : Render (Docker) Base : PostgreSQL managé CI : GitHub
🎯
Pourquoi deux hébergeurs ? Firebase Hosting ne sert que des fichiers statiques (le front Angular compilé). L'API .NET et la base PostgreSQL ont besoin d'un serveur : on les met sur Render (offre gratuite, Docker + base managée). Le front Firebase appelle ensuite l'API Render par son URL publique.

0Préparer le code (« cloud-ready »)

Quelques ajustements pour que l'app fonctionne sur un hébergeur, sans rien casser en local.

  1. API : écouter le port imposé par l'hébergeur. Render fournit une variable PORT. Dans Program.cs : 
    // Render/Railway injectent PORT ; on l'écoute si présent
var listenPort = Environment.GetEnvironmentVariable("PORT");
if (!string.IsNullOrWhiteSpace(listenPort))
    builder.WebHost.UseUrls($"http://0.0.0.0:{listenPort}");
  2. API : accepter une URL PostgreSQL. Render donne une chaîne postgres://user:pass@host/db ; on la convertit au format Npgsql automatiquement (fonction NormalizeConnectionString dans Program.cs).
  3. Fichier render.yaml (racine du repo) : décrit la base + le service web Docker à créer.
  4. Front : firebase.json (dossier public = dist/porteo-web/browser, réécriture SPA) et baseHref="/" dans angular.json.
  5. Front : URL de l'API dans src/environments/environment.prod.ts (renseignée une fois l'API Render en ligne).

ABackend → Render

API .NET 8 (Docker) + base PostgreSQL managée, sur le plan gratuit.

A.1 — Pousser le projet sur GitHub

cd C:\source\Portage
git init
git add .
git commit -m "Porteo - app de portage salarial"
git branch -M main
git remote add origin https://github.com/NeyerBali/Portage.git
git push -u origin main
ℹ️
Le .gitignore exclut déjà bin/, obj/, node_modules/, dist/ : on ne pousse que le code source.

A.2 — Créer les services via le Blueprint

  1. Aller sur render.com → Get Started → Sign up with GitHub (autoriser l'accès au dépôt Portage).
  2. Dashboard Render → New + → Blueprint → connecter le bouton GitHub → choisir NeyerBali/Portage.
  3. Render lit render.yaml et propose : base porteo-db + service web porteo-api (plan Free).
  4. Donner un nom (ex. porteo), branche main, puis Deploy Blueprint.
  5. Render construit l'image Docker (~5–10 min), branche la base, applique les migrations + le seed au 1er démarrage.

A.3 — Récupérer l'URL & tester

Quand porteo-api passe à Live, son URL publique est affichée en haut du service :

RessourceURL
APIhttps://porteo-api.onrender.com
Swaggerhttps://porteo-api.onrender.com/swagger
⏱️
Cold start : sur le plan gratuit, le service s'endort après ~15 min d'inactivité ; la 1ʳᵉ requête suivante prend ~30–50 s. C'est normal.
🧭
L'adresse http://0.0.0.0:10000 vue dans les logs est interne au conteneur (« Now listening on… ») : elle n'est pas accessible depuis le navigateur. On utilise l'URL …onrender.com.

BFrontend → Firebase Hosting

Front Angular compilé en statique, distribué par le CDN Firebase.

B.1 — Installer le CLI Firebase

npm install -g firebase-tools

B.2 — Brancher le front sur l'API Render

Dans crossone.porteo.web.manager/src/environments/environment.prod.ts :

export const environment = {
  production: true,
  apiUrl: 'https://porteo-api.onrender.com/api/'
};

B.3 — Connexion, projet, déploiement

cd C:\source\Portage\crossone.porteo.web.manager

# 1) connexion à ton compte Google
firebase login

# 2) associer TON projet Firebase (le choisir dans la liste, alias "default")
firebase use --add        # → porteo-9cc53

# 3) compiler en production puis déployer
ng build --configuration production
firebase deploy

Résultat :

✔  Deploy complete!
Hosting URL: https://porteo-9cc53.web.app
⚠️
Réécriture SPA = Oui (différent de l'exemple « No ») : l'app utilise le routeur Angular (/missions, /clients/:id…). Sans réécriture vers index.html, ces liens directs renverraient une 404.
📁
Dossier public = dist/porteo-web/browser (Angular 17+ ajoute le sous-dossier browser).

🐞Incident rencontré (et corrigé)

Symptôme : après connexion, la page « ne répond pas » au moment d'afficher le tableau de bord (le login, lui, renvoyait bien 200 OK).

Cause : le menu latéral était généré par un getter renvoyant un nouveau tableau à chaque cycle de détection de changements ; combiné aux routerLink, cela créait une boucle infinie qui figeait le thread principal.

Correctif : menu calculé une seule fois (référence stable), puis nouveau ng build + firebase deploy. Problème résolu. ✅

🔎
Diagnostic clé : le bug se reproduisait aussi dans Edge ⇒ ce n'était pas une extension du navigateur mais bien le code de l'app.
