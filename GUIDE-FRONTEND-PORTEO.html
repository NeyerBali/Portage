Portéo — Guide complet du Front-End Angular

Tout ce qu'il faut comprendre du code Angular du projet pour répondre aux questions de la prof. Lis-le dans l'ordre : chaque section est courte et va à l'essentiel.

0 À lire en premier (le pitch en 60 s)

Portéo est une application de portage salarial : une agence place des consultants chez des clients sur des missions, puis facture les clients et paie les consultants.

Le projet est en 2 parties :

  • Front-end Angular 19 (ce guide) — dossier crossone.porteo.web.manager, hébergé sur Firebase Hosting.
  • Back-end .NET 8 / PostgreSQL (API REST) — dossier crossone.porteo.erp, hébergé sur Render.

Le front ne contient aucune logique métier serveur : il appelle l'API REST en HTTP (JSON) et affiche les résultats. Toute la sécurité réelle est côté back (JWT, rôles, propriété des données).

Comptes de démo (mot de passe Porteo2026!) : neyerbali6@gmail.com (admin) · neyerbali6+consultant@gmail.com (consultant). La connexion passe par une double authentification (code reçu par e-mail).

3 Routing & Guards

Le routeur Angular associe une URL à un composant. Les guards protègent l'accès.

// app-routing.module.ts — routes racine
const routes: Routes = [
  { path: '', pathMatch: 'full', component: VitrineComponent },        // accueil public
  { path: 'auth', loadChildren: () => PublicModule },                // lazy
  { path: '', canActivate:[authGuard], loadChildren: () => PrivateModule }, // lazy + protégé
  { path: '**', redirectTo: '' },
];

Les 3 guards (ce sont des CanActivateFn — fonctions, pas des classes)

GuardRôle
authGuardBloque les pages privées si non connecté → redirige vers /auth/login?returnUrl=….
loginGuardEmpêche d'afficher le login si déjà connecté → redirige vers /dashboard.
roleGuardRéserve certaines routes à l'admin (ex. Clients, Consultants) via data:{roles:['Admin']}.
// auth.guard.ts
export const authGuard: CanActivateFn = (_route, state) => {
  const auth = inject(AuthService); const router = inject(Router);
  if (auth.isLoggedIn()) return true;
  return router.createUrlTree(['/auth/login'], { queryParams:{ returnUrl: state.url } });
};

4 Authentification & double authentification (2FA)

La connexion se fait en 2 étapes (sécurité renforcée) :

  1. Identifiants → l'API renvoie un « défi 2FA » (pas encore de jeton).
  2. Code à 6 chiffres (reçu par e-mail) ou code d'application d'authentification (TOTP) → l'API renvoie le jeton JWT → session ouverte.

AuthService (cœur de l'auth)

  • Garde l'utilisateur courant dans un BehaviorSubject<AuthResult> (flux observable) → la sidebar/topbar se mettent à jour automatiquement.
  • Le jeton JWT est stocké en localStorage via TokenService (qui sait lire l'exp pour vérifier la validité).
  • Getters pratiques : isAdmin, role, consultantId, fullName.
login(payload): Observable<TwoFactorChallenge> { ... }        // étape 1
verifyCode(email, code): Observable<AuthResult> {              // étape 2 (email)
  return this.http.post<AuthResult>(url, {email,code})
    .pipe(tap(r => this.store(r)));   // stocke jeton + user, pousse dans le BehaviorSubject
}

Les 2 intercepteurs HTTP (très demandés à l'oral)

IntercepteurRôle
JwtInterceptorAjoute l'en-tête Authorization: Bearer <token> à chaque requête sortante.
ErrorInterceptorGère les erreurs de façon centralisée : 401 → déconnexion + toast, 403 → « accès refusé », 0 → « serveur injoignable », sinon toast d'erreur.

Ils sont enregistrés dans AppModule via HTTP_INTERCEPTORS avec multi:true (on peut en empiler plusieurs).

5 Communication avec l'API (HttpClient + RxJS)

Tous les appels passent par une façade ApiService qui regroupe un service par domaine :

// api.service.ts
@Injectable({ providedIn: 'root' })
export class ApiService {
  constructor(
    public missions: MissionApiService,
    public clients: ClientApiService,
    public factures: FactureApiService,
    public dashboard: DashboardApiService,
    public justificatifs: JustificatifApiService,
    public alerts: AlertApiService,
    public rh: RhApiService,
    public config: ConfigApiService,
  ) {}
}

Chaque service domaine utilise HttpClient et renvoie un Observable typé :

list(q): Observable<PagedResult<Mission>> {
  let params = new HttpParams();
  Object.entries(q).forEach(([k,v]) => { if (v != null) params = params.set(k, String(v)); });
  return this.http.get<PagedResult<Mission>>(this.base, { params });
}
Observable vs Promise : un Observable est un flux (peut émettre plusieurs valeurs, annulable, composable avec des opérateurs RxJS). On s'y « abonne » avec .subscribe(), ou avec le pipe async dans le template (qui se désabonne tout seul).

Opérateurs RxJS utilisés : debounceTime (recherche), switchMap (annule la requête précédente), mergeMap (laisse passer en parallèle), map, catchError, tap, filter, takeUntil (désabonnement).

6 NgRx — gestion d'état (sur les Missions)

NgRx applique le pattern Redux à Angular : un état centralisé, modifié uniquement par des actions, via des reducers (purs), les effets de bord (HTTP) étant dans des effects. On l'a mis sur les missions pour démontrer la maîtrise (le reste utilise des services classiques, plus simples).

BriqueRôleExemple
Action« Un événement »MissionActions.load({params})
ReducerFonction pure état+action→nouvel étatmet loading=true, range les missions
EffectEffets de bord (HTTP), écoute une action et en émet une autreload$ appelle l'API puis loadSuccess
SelectorLit une tranche de l'état (mémoïsé)selectMissions, selectMissionsTotal
EntityAdapterStocke une collection normalisée + helpersadapter.setAll(...)
// effect : recharge la liste, annule la requête précédente (switchMap)
load$ = createEffect(() => this.actions$.pipe(
  ofType(MissionActions.load),
  switchMap(({params}) => this.api.missions.list(params).pipe(
    map(result => MissionActions.loadSuccess({result})),
    catchError(err => of(MissionActions.loadFailure({error: err.message}))))
)));

Le composant ne fait que dispatcher et sélectionner :

missions$ = this.store.select(selectMissions);   // flux affiché avec | async
reload() { this.store.dispatch(MissionActions.load({ params: this.params })); }
Pourquoi switchMap pour le chargement et mergeMap pour create/update/delete ? Pour une recherche/liste, seule la dernière requête compte → switchMap annule la précédente. Pour des mutations indépendantes, on veut toutes les laisser aboutir → mergeMap.

7 Formulaires réactifs (Reactive Forms)

On utilise les formulaires réactifs (le modèle est défini en TypeScript, pas dans le HTML) — plus testables et puissants que les « template-driven ».

form = this.fb.group({
  email:    ['', [Validators.required, Validators.email]],
  password: ['', [Validators.required]],
  remember: [true],
});
get f() { return this.form.controls; }      // accès aux contrôles dans le template
submit() { if (this.form.invalid) { this.form.markAllAsTouched(); return; } /* ... */ }

Côté template : [formGroup]="form", formControlName="email", et affichage des erreurs avec *ngIf="f.email.touched && f.email.errors?.['required']". Exemples concrets : login, popups mission/client/consultant/facture, justificatif, profil/sécurité.

8 Les graphiques ★ (100 % SVG fait main, sans librairie)

Point fort à mettre en avant : les graphiques ne dépendent d'aucune librairie (pas de Chart.js). Ce sont des composants Angular qui dessinent du SVG à partir des données, dans shared/components/charts/. Ça montre la maîtrise du SVG + du data-binding.

① Courbe d'aire — app-area-chart (CA sur 12 mois)

On calcule, dans ngOnChanges, les coordonnées (x,y) de chaque point : x régulier, y proportionnel à la valeur / max. On construit :

  • un chemin ligne M x y L x y L … (move-to / line-to),
  • un chemin aire = la ligne refermée vers le bas (… L xN bas L x0 bas Z) rempli d'un dégradé <linearGradient>,
  • des cercles invisibles (rayon 16) au-dessus de chaque point pour afficher une tooltip (mois + montant) au survol.
const max = Math.max(1, ...pts.map(p => p.value));
const coords = pts.map((p,i) => ({ x: pad + i*stepX,
                                   y: H - pad - (p.value/max)*(H - pad*2) }));
this.linePath = coords.map((c,i) => `${i?'L':'M'} ${c.x} ${c.y}`).join(' ');
this.areaPath = `${this.linePath} L ${last.x} ${H-pad} L ${first.x} ${H-pad} Z`;

② Donut — app-donut-chart (missions par statut)

Astuce classique : un seul <circle> par segment, coloré via stroke-dasharray (longueur du trait = part du segment sur la circonférence 2πr) et décalé via stroke-dashoffset. Le tout tourné de -90° pour démarrer en haut.

const C = 2*Math.PI*46;                  // circonférence (r=46)
const len = (segment.value/total) * C;   // longueur du segment
{ dash: `${len} ${C-len}`, offset: -acc }   // acc = cumul des segments précédents

③ Barres — app-bar-list (top 5 clients)

Pas de SVG ici : de simples <div> dont la largeur en % = valeur / max, avec un dégradé. Tooltip au survol.

④ Mini-courbe — app-spark (tendance dans les cartes KPI)

Même principe que la courbe d'aire mais en miniature (84×30 px), normalisée entre min et max : y = H - ((v-min)/(max-min))·H.

Les couleurs viennent des variables CSS (var(--primary), var(--accent)) → les graphiques changent de couleur avec le thème/palette, automatiquement.

9 Thème & design system

Tout le style repose sur des variables CSS (design tokens) définies dans src/styles/_tokens.scss : couleurs, espacements, rayons, ombres, typographies.

  • Mode sombre : attribut [data-theme="dark"] sur <html> qui redéfinit les variables.
  • 5 palettes de marque ([data-palette="azur"], gold, violet, coral, emerald) qui redéfinissent l'échelle de couleur. Azur (bleu ciel + jaune) est la palette par défaut.
  • ThemeService applique ces attributs et mémorise le choix dans localStorage. Un petit script dans index.html applique le thème avant le rendu (pas de « flash »).

Le menu et tous les composants utilisent ces variables → un seul endroit à changer pour reskins. Les icônes du menu sont des SVG servis depuis public/icons/ via des classes CSS .icon.icon-xxx.

Zoom — le tableau générique (le plus « Angular »)

Un seul composant <app-data-table> gère tri + recherche + pagination pour n'importe quelles données. Le parent fournit les colonnes en @Input et, pour les cellules spéciales (badge de statut, boutons d'action), des gabarits projetés :

  <ng-template [appColumnCell]="'statut'" let-f>
     <app-status-badge [status]="f.statut"></app-status-badge>
  </ng-template>
</app-data-table>

En interne, le tableau récupère ces gabarits via @ContentChildren(ColumnCellDirective) et les affiche avec *ngTemplateOutlet. C'est la projection de contenu + la réutilisabilité.

12b Assistant IA & Parallélisme ★ (le bonus qui impressionne)

🤖 Assistant IA (Groq)

Page Administration ▸ Assistant IA. Le front n'appelle jamais Groq directement (la clé resterait exposée) : il appelle notre API (/api/assistant/ask), qui :

  1. construit un contexte à partir des vraies données (factures impayées + retards, missions en cours, CA) ;
  2. l'injecte dans le prompt et interroge Groq (modèle llama-3.3-70b-versatile, API compatible OpenAI) ;
  3. renvoie la réponse. La clé Groq est une variable d'environnement serveur (jamais dans le bundle).

On peut ainsi : lister les impayés, générer un e-mail de relance, repérer les missions à risque, obtenir une synthèse. Bonus : résumé IA d'une mission (bouton sur la fiche mission).

Streaming mot-à-mot (Server-Sent Events → Observable RxJS)

L'endpoint /api/assistant/ask-stream demande à Groq un flux (stream:true) et réécrit les tokens au fil de l'eau (Response.WriteAsync + FlushAsync). Côté Angular, on lit ce flux avec fetch + ReadableStream, enveloppé dans un Observable qui émet token par token :

askStream(question): Observable<string> {
  return new Observable(sub => {
    fetch(url, { method:'POST', headers:{Authorization:`Bearer ...`}, body }).then(async res => {
      const reader = res.body.getReader(); const dec = new TextDecoder();
      for(;;){ const {done,value}=await reader.read(); if(done) break; sub.next(dec.decode(value)); }
      sub.complete();
    }).catch(e => sub.error(e));
  });
}

Le composant ajoute chaque chunk au message en cours (curseur clignotant). Repli automatique sur la réponse complète si le flux est indisponible.

⚡ Parallélisme & concurrence (page « Centre d'analyse »)

Page Administration ▸ Centre d'analyse. Elle démontre 3 mécanismes distincts :

1) Web Worker — vrai parallélisme (multi-thread)

Une simulation Monte-Carlo (≈20 000 tirages) du recouvrement des impayés tourne sur un thread séparé via un Web Worker. Le thread principal (UI) ne se fige pas : la barre de progression bouge, la page reste fluide.

// création + communication
const w = new Worker(new URL('./montecarlo.worker', import.meta.url));
w.onmessage = ({data}) => { if(data.type==='progress') this.progress=data.value; ... };
w.postMessage({ invoices, runs: 20000 });
// dans le worker (.worker.ts) : boucle de calcul + postMessage(progress/done)
C'est LE point « parallèle » : JavaScript est mono-thread sur l'UI ; un Web Worker offre un vrai second thread du navigateur pour les calculs lourds, sans bloquer le rendu.

2) forkJoin — requêtes HTTP en parallèle

Au chargement, missions + factures + consultants sont demandés simultanément et la suite ne s'exécute qu'une fois tous arrivés (vs en série, bien plus lent) :

forkJoin({ missions: api.missions.list(...), factures: api.factures.list(), consultants: api.consultants.list() })
  .subscribe(({missions,factures,consultants}) => { /* tout est prêt */ });

3) combineLatest — combinaison réactive de flux

Deux flux (recherche + statut) sont combinés en continu : à chaque changement de l'un, la liste filtrée se recalcule.

filtered$ = combineLatest([factures$, search$, statut$]).pipe(
  map(([list, q, st]) => list.filter(f => (!st || f.statut===st) && matches(f, q))));

+ le streaming IA ci-dessus est une 4ᵉ forme de concurrence (un flux asynchrone consommé au fil de l'eau).

15 ★ Questions probables de la prof — réponses prêtes

Pourquoi NgModule et pas le mode « standalone » ?
Choix d'architecture pour reproduire le projet d'entreprise de référence. En NgModule, on déclare les composants dans des modules (declarations) et on partage via un SharedModule. Le tsconfig est réglé en conséquence (useDefineForClassFields:false).
C'est quoi le lazy loading, et où l'utilisez-vous ?
Le chargement à la demande : le code d'un module n'est téléchargé que lorsqu'on visite ses routes. On l'utilise pour PublicModule (/auth) et PrivateModule (l'app connectée) via loadChildren. Ça réduit le bundle initial.
Observable vs Promise ? Pourquoi RxJS ?
Une Promise émet une seule valeur et n'est pas annulable. Un Observable est un flux (0..n valeurs), annulable et composable (opérateurs). Idéal pour HTTP, recherche en temps réel (debounceTime+switchMap) et le temps réel. On s'abonne avec subscribe ou le pipe async.
switchMap vs mergeMap ?
switchMap annule l'Observable interne précédent (parfait pour une recherche/liste : on ne veut que le dernier résultat). mergeMap exécute tout en parallèle (parfait pour des créations/suppressions indépendantes).
Expliquez NgRx : store, action, reducer, effect, selector.
Store = état unique en lecture seule. Action = un événement (ex. load). Reducer = fonction pure qui produit le nouvel état. Effect = gère les effets de bord (appel HTTP) en écoutant une action et en émettant une autre. Selector = lit/dérive une partie de l'état (mémoïsé). On l'a mis sur les missions ; on utilise EntityAdapter pour stocker la collection proprement.
Pourquoi NgRx seulement sur les missions et pas partout ?
NgRx ajoute de la complexité : il est justifié pour un état partagé/complexe (la liste paginée des missions, rechargée après mutation). Pour des écrans plus simples, des services + Observables suffisent. C'est un choix d'architecture pragmatique.
À quoi servent les intercepteurs HTTP ?
Intercepter toutes les requêtes/réponses. JwtInterceptor ajoute le jeton Bearer ; ErrorInterceptor centralise la gestion d'erreur (401 → déconnexion, etc.). Enregistrés avec HTTP_INTERCEPTORS + multi:true.
Comment sécurisez-vous les routes ?
Avec des guards (CanActivateFn) : authGuard (connecté ?), loginGuard (déjà connecté → dashboard), roleGuard (admin uniquement). ⚠️ C'est une sécurité d'UX ; la vraie sécurité est côté API (JWT + rôles + filtre de propriété des données).
Reactive Forms vs Template-driven ?
On utilise les Reactive Forms : le modèle (FormGroup) est défini en TypeScript, validations explicites (Validators), plus testable et adapté aux formulaires complexes. Template-driven = tout dans le HTML avec ngModel, plus simple mais limité.
Comment évitez-vous les fuites mémoire (subscriptions) ?
On se désabonne : takeUntil(destroy$) dans ngOnDestroy, ou le pipe async (auto-désabonnement), ou on stocke la Subscription et on l'unsubscribe (ex. SignalR dans le layout).
Quels hooks de cycle de vie utilisez-vous ?
ngOnInit (init/chargement), ngOnDestroy (nettoyage), ngOnChanges (recalcul des graphiques quand les données changent), ngAfterViewInit (accès au DOM : <canvas> de signature, animations de la vitrine).
Comment sont faits les graphiques ? Quelle librairie ?
Aucune librairie : des composants Angular qui génèrent du SVG. La courbe = chemins path (line-to) ; le donut = stroke-dasharray/dashoffset sur un cercle ; les barres = largeur CSS en %. Les couleurs viennent des variables CSS du thème.
@Input / @Output / @ViewChild / @ContentChildren ?
@Input = données du parent vers l'enfant ; @Output = événement de l'enfant vers le parent (EventEmitter) ; @ViewChild = référence à un élément/composant de mon template (ex. canvas) ; @ContentChildren = éléments projetés par le parent (ex. les gabarits de cellules du tableau générique).
Pipe pur vs impur ? Donnez un exemple.
Pur (défaut) : recalculé seulement si l'entrée change → performant. Impur : recalculé à chaque détection. Notre pipe tr (traduction) est impur car son résultat dépend de la langue courante (état externe).
C'est quoi l'injection de dépendances ici ?
Les services sont @Injectable({providedIn:'root'}) (singletons), injectés via le constructeur des composants/services. Ex. : constructor(private api: ApiService, public auth: AuthService). Angular fournit l'instance automatiquement.
Comment gérez-vous le multilingue et le thème ?
Le thème via des variables CSS + attributs [data-theme]/[data-palette] pilotés par ThemeService. Le multilingue via LangService (à base de signal) + un pipe impur tr ; un dictionnaire EN, le FR étant la valeur par défaut.
Comment fonctionne le drag & drop du Kanban ?
Avec le Angular CDK : cdkDropListGroup connecte les colonnes, cdkDrag rend les cartes déplaçables, et à la dépose on appelle transferArrayItem + un PUT à l'API pour changer le statut (avec rollback si erreur).
Comment fonctionne le temps réel ?
SignalR (WebSocket avec repli automatique). Le client (RealtimeService) ouvre une connexion authentifiée (JWT), écoute l'événement notification et le pousse dans un Subject RxJS ; le layout s'y abonne pour afficher un toast et la cloche.
environment.ts, à quoi ça sert ?
À séparer la config dev et prod (notamment apiUrl). Le build de production remplace environment.ts par environment.prod.ts (URL Render).
Comment télécharge-t-on un PDF / un Excel ?
Le PDF facture est généré côté API (QuestPDF) et renvoyé en Blob (responseType:'blob') ; le front crée une URL objet et déclenche le téléchargement. L'Excel est généré côté front avec SheetJS (XLSX.writeFile).
⚡ JavaScript est mono-thread : comment faites-vous du parallélisme ?
Sur l'UI, oui (un seul thread). Pour un vrai calcul parallèle, on utilise un Web Worker (page « Centre d'analyse ») : la simulation Monte-Carlo tourne sur un thread séparé du navigateur, communique par postMessage, et l'UI reste fluide (la barre de progression bouge pendant le calcul).
Web Worker : comment ça communique avec le composant ?
Par messages : worker.postMessage(data) côté composant, addEventListener('message', …) + postMessage(résultat) côté worker, et worker.onmessage pour recevoir. Pas de mémoire partagée : on échange des données (clonées). On crée le worker avec new Worker(new URL('./x.worker', import.meta.url)).
forkJoin vs concat/concatMap (parallèle vs séquentiel) ?
forkJoin lance plusieurs Observables en parallèle et émet quand tous sont terminés (comme Promise.all). concat/concatMap exécute l'un après l'autre (séquentiel). On charge missions+factures+consultants avec forkJoin → plus rapide qu'en série.
combineLatest, à quoi ça sert ?
À combiner plusieurs flux et réagir dès que n'importe lequel change : on combine « recherche » + « statut » pour recalculer la liste filtrée en temps réel. (À distinguer de forkJoin qui attend la fin de tout, et de zip qui apparie les émissions.)
Le streaming de l'IA, c'est de la concurrence ?
Oui : le serveur renvoie un flux de tokens (SSE), consommé au fil de l'eau côté client via fetch+ReadableStream enveloppé dans un Observable qui émet morceau par morceau, pendant que l'UI continue de tourner. C'est de la programmation asynchrone/réactive.
